ROUND 105 — ESTIMATED BMR + BREAKDOWN DISCLAIMER + PROFILE MODAL (2026-06-25)

── PROBLEM ───────────────────────────────────────────────────────────────────
When a user creates a goal but has not filled in height, age, or sex on their
profile, bmrForDate() returned user.bmr which defaults to 0. With bmr=0 the
gauge was completely empty and totalBurn=0, making the calorie breakdown
meaningless even though a goal existed and a starting weight was recorded.

── BMR.TS — estimateBmrFromWeight ────────────────────────────────────────────
Added:
  export function estimateBmrFromWeight(weightKg: number): number
  → Math.round(10 * weightKg + 810)

Derivation: Mifflin–St Jeor (10·w + 6.25·h - 5·a + sex) evaluated at
population-average values (h=170 cm, a=35 yr, sex-neutral midpoint).
  Male:   10w + 1062.5 - 175 + 5   = 10w + 892.5
  Female: 10w + 1062.5 - 175 - 161 = 10w + 726.5
  Avg:    10w + 809.5 ≈ 10w + 810
Result is a reasonable rough estimate (±10–15% for typical adults).

── USEDAY.TS — bmrIsEstimated flag ───────────────────────────────────────────
DayData interface: added bmrIsEstimated: boolean.
Logic:
  const bmrActual = user ? bmrForDate(date, weights, user) : 0;
  const latestWeight = [...weights sorted desc][0] ?? null;
  const bmrIsEstimated = bmrActual === 0 && latestWeight != null;
  const bmr = bmrActual > 0 ? bmrActual
    : latestWeight ? estimateBmrFromWeight(latestWeight.weightKg)
    : 0;
summarizeDay(bmr, ...) now receives the estimated BMR so totalBurn is
non-zero and the gauge renders meaningful numbers.

── WHEELPI CKER: FIX SPLIT-DECIMAL EMPTY+CENTERÁT STATE ─────────────────────
Problem: When a WheelPicker had step<1 (decimal/split mode), an empty value,
AND a centerAt prop, it rendered as a single long unified <select> instead of
the expected two split wheels. This meant "Target weight" always showed as one
long drum-roll list on first open (the field starts empty + has centerAt set to
the current start weight for centering the picker).

Root cause: The `if (isEmpty && centerAt !== undefined)` branch in the split-
decimal path built a single allOptions list. This was designed before split mode
existed and was never updated.

Fix (WheelPicker.tsx):
  - Replace the unified-select empty-state with two split selects that each
    have a "—" placeholder option centred at the respective component of centerAt:
      whole wheel: "—" inserted before the cWhole option
      decimal wheel: "—" inserted before the cDec option
  - When either wheel changes from "—", onChange fires with the combined value
    (newWhole.cDec or cWhole.newDec). Value becomes non-empty → normal split
    mode takes over on next render.
  - Result: Target weight field always shows as two wheels (kg | .g) matching
    the Start weight field, in both Simple and Custom modes.

── DIARY SORT BUG FIX ───────────────────────────────────────────────────────

Root cause: newId() uses crypto.randomUUID() — fully random UUIDs with no
time component. Dexie's byDate() returns entries in primary-key (UUID) order,
which is random. TodayScreen applied .reverse() on this random array, so the
order of logged items was unpredictable.

The FoodEntry.createdAt field (ISO timestamp string) has always been set at
log time but was never used for ordering.

Fix: in useDay.ts, after the Dexie query, sort rawFoods by createdAt ascending
before returning. The existing .reverse() in TodayScreen then correctly yields
newest-first.

  const foods = [...rawFoods].sort((a, b) =>
    (a.createdAt ?? '').localeCompare(b.createdAt ?? ''),
  );

── TITLE SPACING ─────────────────────────────────────────────────────────────

"Diary" title container: mb-4 (16px) → mb-2 (8px).
Combined with WeekStrip's pt-2 (8px), visual gap from title to day letters
is now ~16px (was ~24px).

── NEGATIVE CALORIE DISPLAY ──────────────────────────────────────────────────

Motivation: the gauge showed "500" for both "500 left" (good) and "500 over"
(bad) — the number had no polarity signal. Food entries in the diary also
showed calories as neutral positive numbers.

GAUGE CENTER (when over):
  The number display is split into a styled minus sign + the absolute value:

    <div
      className="flex items-center gap-1"
      style={left < 0 ? { transform: 'translateX(-4px)' } : undefined}
    >
      {left < 0 && (
        <span className="text-title font-semibold leading-none text-content">−</span>
      )}
      <span className="text-hero font-semibold leading-none tracking-tight text-content">
        {Math.abs(left)}
      </span>
    </div>

  - text-title = 22px (closest token to 24px; text-display at 30px is too large).
  - gap-1 = 4px spacing between − and the number.
  - translateX(-4px) shifts the container left only when over, so the number
    part stays optically centered in the arc regardless of state.
  - When on target (left ≥ 0): no minus sign, no offset — unchanged.

FOOD KPI TILE (bottom of gauge card):
  Both the hasTarget and no-goal variants updated:
    `−${Math.round(consumed)}` instead of `Math.round(consumed)`
  Shows "—" when nothing logged (unchanged).

── SLIDER THUMB REACTIVITY ──────────────────────────────────────────────────
Problem: Previous implementation animated scale-up on press. On iOS WKWebView,
the 200ms animation meant the thumb only started scaling after the user had
already begun dragging, making it feel unresponsive.

Fix (src/index.css):
  input[type="range"]::-webkit-slider-thumb {
    transition: transform 200ms cubic-bezier(0.34, 1.56, 0.64, 1);
  }
  input[type="range"]:active::-webkit-slider-thumb {
    transform: scale(2);
    transition: transform 0ms;       /* ← instant scale-up on press */
  }

Scale-up is now instant (0ms) so the thumb expands the moment finger lands.
Scale-down uses the 200ms spring so the release feels fluid.
